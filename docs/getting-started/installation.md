# Minder Platform - Installation Guide

## Prerequisites

### Required Software

- **Docker** 20.10+
- **Docker Compose** 2.20+
- **Git** (for development)

### System Requirements

- **RAM**: 8GB minimum, 16GB recommended
- **Disk**: 20GB free space (+3GB for AI models)
- **CPU**: 4 cores minimum, 8 cores recommended
- **OS**: Linux, macOS, or Windows with WSL2
- **Network**: Stable internet for AI model downloads (first run only)

## Installation Methods

### Method 1: Automated Setup (Recommended)

The `setup.sh` script provides complete lifecycle management for the Minder platform.

```bash
# Clone the repository
git clone git@github.com:wish-maker/minder.git
cd minder

# Install and start everything
bash setup.sh install
# (or, if already installed:)
bash setup.sh start
```

That's it. The platform provisions itself with:

**✨ AUTOMATIC FEATURES:**
- 🤖 **AI Models:** Auto-pulls the models listed in `OLLAMA_PULL_MODELS` (default llama3.2 + nomic-embed-text) when the internal Ollama container runs
- 🔐 **Secrets:** Fills any `CHANGEME` values in `./.env` with generated secrets (JWT auth; passwords for Postgres/Redis/Neo4j/etc.)
- 🗄️ **Databases:** Creates and initializes all required databases
- 🌐 **Networking:** Sets up Docker networks automatically
- 🔁 **Env mirror:** Copies root `./.env` → `docker/compose/.env` (the path Compose reads)

### Method 2: Manual Setup

> Manual setup bypasses the conveniences `setup.sh` gives you (secret auto-fill, env mirroring, model pulling, migrations). Prefer Method 1 unless you have a reason not to.

#### Step 1: Environment Configuration

Root `./.env` is the **single source of truth**. `docker/compose/.env` is auto-generated by `setup.sh` — do not edit it directly.

```bash
# Create the environment file
cp .env.example .env

# Fill in the CHANGEME secrets yourself
nano .env

# Mirror to the path Compose reads (setup.sh does this for you on start/restart)
cp .env docker/compose/.env
```

#### Step 2: Start Services

```bash
# Start all services (uses the hand-maintained compose file)
docker compose --file docker/compose/docker-compose.yml up -d
```

Compose defines 31 services; a full `setup.sh install` runs 30 (see caveats below), grouped as:
- Reverse Proxy (1): Traefik v3 (Authelia is defined but currently commented out / disabled)
- Storage (7, internal-only): PostgreSQL, Redis, Qdrant, Neo4j, MinIO, RabbitMQ, Apicurio schema-registry
- Inference (2): Ollama (internal-only), OpenWebUI (via Traefik)
- Core APIs (8): API Gateway, Plugin Registry, Marketplace, Plugin State Manager, RAG Pipeline, Model Management, TTS/STT, Graph-RAG
- Observability (7): Prometheus, Grafana, Alertmanager, InfluxDB, Telegraf, Jaeger, OTel Collector
- Metrics Exporters (6): postgres/redis/rabbitmq/node exporters, cAdvisor, blackbox-exporter

#### Step 3: Verify Installation

Only app and monitoring services are exposed on host ports. Storage backends and Ollama are internal-only (reached over the Docker network or via Traefik).

```bash
# Core API health (host-exposed 8000-8006, 8008)
curl http://localhost:8000/health  # API Gateway
curl http://localhost:8001/health  # Plugin Registry
curl http://localhost:8002/health  # Marketplace
curl http://localhost:8003/health  # Plugin State Manager
curl http://localhost:8004/health  # RAG Pipeline
curl http://localhost:8005/health  # Model Management
curl http://localhost:8006/health  # TTS/STT Service
curl http://localhost:8008/health  # Graph-RAG

# Monitoring (host-exposed)
curl http://localhost:9090/-/healthy  # Prometheus
curl http://localhost:3000/api/health # Grafana

# Ollama is internal-only — check it from inside the container:
docker exec minder-ollama ollama list
```

## Lifecycle Management

The `setup.sh` script provides comprehensive enterprise lifecycle management capabilities:

### Available Commands

The full command set is:

```
install | start | stop | restart | status | logs | shell | migrate |
backup | restore | doctor | update | ollama-mode | sync-postgres-password | uninstall
```

```bash
# Installation & service management
bash setup.sh install                # Install and start
bash setup.sh start                  # Start all services
bash setup.sh stop                   # Stop all services
bash setup.sh restart                # Restart all services

# Status & logs
bash setup.sh status                 # Show service status
bash setup.sh logs                   # View service logs (all)
bash setup.sh logs <service>         # View specific service logs

# Operations
bash setup.sh shell <service>        # Interactive shell in container
bash setup.sh migrate                # Run Alembic DB migrations
bash setup.sh doctor                 # Deep diagnostics (disk, ports, secrets, images, versions)
bash setup.sh ollama-mode <mode>     # Switch Ollama between internal / external
bash setup.sh sync-postgres-password # Re-sync the Postgres password across services

# Data management
bash setup.sh backup                 # Full backup (Postgres, Neo4j, InfluxDB, Qdrant, .env)
bash setup.sh restore <archive>      # Restore from backup

# Updates & uninstall
bash setup.sh update                 # Update Docker images
bash setup.sh uninstall              # Remove the platform
```

### Service Status

The `status` command provides a comprehensive overview:

```bash
./setup.sh status
```

Output includes:
- Running containers with health status
- Port mappings
- Resource usage (CPU, memory)
- Volume information
- Network status

### Update Management

The platform supports Docker image updates:

```bash
# Update images
bash setup.sh update
```

The update process:
1. Checks configured images for newer versions
2. Pulls updated images
3. Rebuilds custom Minder images
4. Prompts for service restart

### Backup & Recovery

Automated backup functionality:

```bash
# Create backup
./setup.sh backup

# Backup includes:
# - Environment configuration (.env file)
# - All databases (PostgreSQL dump)
# - Service configurations
```

Backups are stored in `backups/minder-backup-YYYYMMDD-HHMMSS/` with:
- `env.backup` - Environment configuration
- `databases.sql` - Complete database dump

## Startup

First startup pulls images and (in internal Ollama mode) downloads the models in
`OLLAMA_PULL_MODELS` (~3GB by default), so allow extra time on the first run.
On an ARM Raspberry Pi 4, initial provisioning takes several minutes.

Rough order:
1. Storage backends (PostgreSQL, Redis, Qdrant, Neo4j, MinIO, RabbitMQ, schema-registry)
2. Traefik reverse proxy
3. Core APIs (Gateway, Registry, Marketplace, State Manager, RAG, Model Management, TTS/STT, Graph-RAG)
4. Ollama + model pulls (internal mode only)
5. Observability stack (Prometheus, Grafana, InfluxDB, Jaeger, exporters) + OpenWebUI

**Final state (via `setup.sh install`):** 30 containers running. Of the healthchecked
services, 3 (otel-collector, redis-exporter, rabbitmq-exporter) have no healthcheck by
design — they show as "no-healthcheck", not "unhealthy".

**Known caveats (verified on a 2026-07-10 clean install):**
- `schema-registry` is defined in compose but not wired into `setup.sh` → it does **not**
  start ([#42](https://github.com/wish-maker/minder/issues/42)).
- MinIO is initialized only by `install`; `bash setup.sh start` alone omits it
  ([#43](https://github.com/wish-maker/minder/issues/43)).
- On Docker Desktop, `telegraf` crash-loops (docker.sock permission) and stays unhealthy,
  adding a 120s wait to install/start ([#40](https://github.com/wish-maker/minder/issues/40)).
- The `Running migrations` step warns for all services — the images have no alembic and
  services self-init their schema; the warnings are harmless
  ([#41](https://github.com/wish-maker/minder/issues/41)).

## Troubleshooting

### Port Conflicts

If you have port conflicts, edit the ports in `docker/compose/docker-compose.yml`:

```yaml
ports:
  - "8080:8000"  # Change API Gateway to port 8080
```

### Memory Issues

If you experience memory issues, increase Docker memory limit:

- **Docker Desktop**: Settings → Resources → Memory (8GB+)
- **Linux**: No action needed

### Database Connection Errors

```bash
# Using lifecycle script
./setup.sh stop
./setup.sh start

# Or manual reset
docker compose -f docker/compose/docker-compose.yml down -v
docker compose -f docker/compose/docker-compose.yml up -d postgres
```

### Services Not Starting

```bash
# Check service status
./setup.sh status

# View logs for issues
./setup.sh logs

# Restart specific service
docker compose -f docker/compose/docker-compose.yml restart <service>
```

### Docker Image Issues

```bash
# Update images
bash setup.sh update

# Force rebuild
docker compose --file docker/compose/docker-compose.yml build --no-cache <service>
```
